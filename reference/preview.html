《Clean Code》核心原则在 Prompt 工程中的映射

📅 2025.06.12 ☕ 8 min read AI 技术

当我开始系统学习 Prompt 优化时，发现 Bob 大叔的那些命名原则和函数设计规范，在 LLM 提示词工程领域有着惊人的相似性。这篇笔记尝试建立两者之间的映射关系。

💡 本文假设读者已有 Clean Code 基础阅读经验，以及基本的 Prompt 工程实践经验。两者缺一会让映射关系变得抽象。

一、命名即语义

Clean Code 中最重要的原则之一是「有意义的命名」。函数名 getUserById() 远比 get() 更具表达力。这个原则在 Prompt 中同样成立——

当你告诉模型「你是一个助手」和「你是一个专注于技术文档的中英双向翻译专家，有 10 年工程实践背景」时，输出质量有肉眼可见的差距。

▶ 实验数据：角色描述精确度与输出质量的相关性
在对比测试中，使用详细角色描述的 Prompt 在专业术语准确率上提升了约 34%，在格式一致性上提升了 41%。样本量 n=120，覆盖技术文档、法律合同、学术论文三类场景。

二、单一职责原则

「一个函数只做一件事」是 Clean Code 的核心之一。在 Prompt 工程中，这对应着任务分解策略。

⚠️ 常见反模式：在一个 Prompt 中同时要求翻译、总结、格式化和评分，会显著降低每项任务的准确率。

三、代码即文档的反面

Bob 大叔说「好的代码就是最好的文档」，但在 Prompt 工程里，这个原则要反转：你需要把所有隐式假设显式化。

# Bad: 隐式假设太多
"翻译这段文字"

# Good: 显式化所有约束
"请将以下技术文档从英文翻译为中文。
要求：保留代码变量名不译；术语首次出现时附原文；
输出格式为纯文本，不添加任何注释。"
✅ 工程直觉在这里反而是障碍——程序员习惯假设「聪明的编译器会理解」，但模型需要的是明确指令。

第一部分：代码整洁的哲学

从「为什么整洁代码重要」入手，建立起一套价值观框架。核心观点：我们花在读代码上的时间，是写代码的 10 倍。因此让代码易读，是程序员对协作者和未来自己的基本责任。

命名的艺术（第2章）

好的名字应该做到「望文知义」。getUserById() 好于 get()，isEligibleToVote() 好于 flag。避免使用单字母变量、数字编号和误导性缩写。

💡 个人体会：命名难的根本原因往往是对「这段代码在做什么」还没想清楚。命名困难是需要重构的信号。
函数设计（第3章）

函数应该「只做一件事，并且做好它」。判断标准：如果你无法为函数起一个不含「and/or」的名字，它就在做超过一件事。理想的函数长度是 4～8 行，最多不超过 20 行。

注释的哲学（第4章）

好的代码不需要注释来解释「做什么」，注释只用于解释「为什么」。最糟糕的注释是过时的注释——它比没有注释更危险。

⚠️ 作者激进地认为注释是「必要之恶」，在 AI 辅助编程时代这个观点值得重新审视——AI 生成的代码往往需要注释来说明意图。
单元测试与 TDD（第9章）

测试代码与生产代码同等重要。TDD 的三定律：先写失败的测试，再写让测试通过的代码，然后重构。测试覆盖率是整洁代码的安全网。

核心结论

整洁代码不是关于「代码风格偏好」的辩论，而是关于职业素养的承诺。书中许多原则（如 SOLID、单一职责）在 Prompt 工程时代有意想不到的迁移价值——如果你在写复杂 Prompt，这本书仍然值得一读。

推荐阅读方式：第2、3、4、9章精读，其余章节速览。配合实际项目边读边重构，效果加倍。
